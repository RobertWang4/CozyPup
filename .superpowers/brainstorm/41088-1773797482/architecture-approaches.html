PetCare Agent — 架构方案对比

三种 Router + Agent 的实现方式，核心区别在于意图识别和 Agent 编排

A

LLM Router（推荐）

意图识别：用便宜模型（GPT-4o-mini / Haiku）做 Router，一次调用返回 intent + entities JSON

Agent 编排：Python 代码根据 intent 分发到对应处理函数，每个 Agent 是一个独立模块

对话模型：强模型（GPT-4o / Sonnet）只在 Consult Agent 里用

Pros

  • 灵活，新意图只需改 prompt
  • 意图识别准确率高
  • 便宜模型调用成本极低
  • 架构简单，代码量少

Cons

  • 每次多一次 LLM 调用（Router）
  • Router 延迟 ~300ms
B

关键词 + 规则 Router

意图识别：用正则/关键词匹配（"记录"→Record，"附近"→Map，其余→Consult）

Agent 编排：同 A，代码分发

对话模型：同 A

Pros

  • 零额外 LLM 成本
  • 响应更快，无 Router 延迟
  • 实现最简单

Cons

  • 自然语言理解差，容易误判
  • "今天狗狗吐了"→ 记录还是问诊？规则很难判断
  • 每加新意图都要写规则，维护成本高
C

单模型 + Function Calling

意图识别：不单独做 Router，对话模型直接通过 function calling / tool use 决定调用哪个能力

Agent 编排：模型自己选择调用 record()、consult()、search_map() 等函数

对话模型：全程强模型

Pros

  • 最自然的交互，模型自己判断该做什么
  • 支持一句话触发多个动作（"狗狗吐了，帮我记一下，附近有急诊吗"）
  • 架构最优雅

Cons

  • 全程用强模型，成本最高
  • 调试难度大，行为不如显式路由可控
  • 对 prompt 工程要求高
